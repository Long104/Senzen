#### 2.3 Fixed fetch/client.tsx — JSX Returns on Error

**File**: `src/fetch/client.tsx`

```ts
// BEFORE (broken — returns a React div element where callers expect JSON data):
export async function fetchGet(url: string): Promise<any> {
  try {
    const res = await fetch(...)
    if (!res.ok) throw new Error("cannot fetch data");
    return await res.json();
  } catch (error) {
    console.error(error);
    return <div>Error loading data</div>;  // ← JSX in a data function!
  }
}

// AFTER (fixed — throws error so callers can handle it):
export async function fetchGet(url: string): Promise<any> {
  const res = await fetch(...)
  if (!res.ok) {
    throw new Error(`Failed to fetch ${url}: ${res.statusText}`);
  }
  return await res.json();
}
```

This fix was applied to all 4 functions: `fetchGet`, `fetchPost`, `fetchUpdate`, `fetchDelete`.

**Why this was broken**: The function returns `Promise<any>` but was returning `<div>` (a React element) in the catch block. Callers like `api/Plan.tsx` would receive a React element instead of JSON, then pass it to React Query. This caused `Objects are not valid as a React child` errors.

#### 2.4 Fixed fetch/server.tsx — Same JSX Issue

**File**: `src/fetch/server.tsx`

Same fix as above. The server-side fetch function also returned `<div>Error loading data</div>` on error.

#### 2.5 Fixed Auth Context Typo

### Phase 2: High Priority Fixes

#### 2.7 Stable QueryClient

**File**: `src/app/(product)/layout.tsx`

```ts
// BEFORE (broken — new instance every render, cache lost constantly):
export default function RootLayout({ children }) {
  const queryClient = new QueryClient({ ... });
  // ...
}

// AFTER (fixed — stable instance via useState initializer):
export default function RootLayout({ children }) {
  const [queryClient] = useState(() => new QueryClient({ ... }));
  // ...
}
```

**Why this was broken**: `new QueryClient()` in the component body creates a fresh instance on every re-render. React Query stores all cached data (plans, categories, transactions) inside the QueryClient instance. Each re-render destroyed the cache, forcing re-fetches of everything.

## Commit 3: `f78e5f6` — Middleware Redirect Loop Fix

### What Was Wrong

The middleware (`src/middleware.ts`) created an infinite redirect loop when the JWT cookie was expired:

```
Step 1: User requests /home with an expired JWT cookie
Step 2: Middleware decodes JWT → finds exp is in the past
Step 3: Calls handleLogout() which does fetch("/api/logout") to backend
Step 4: Backend is down (or unreachable) → catch block fires
Step 5: catch block returns redirect("/sign-in")
         BUT the JWT cookie is NEVER cleared
Step 6: Browser follows redirect to /sign-in
         JWT cookie is still present (still expired)
Step 7: Middleware sees token on /sign-in
         Path "/sign-in" is in redirectToHome list → redirects to /home
Step 8: Back to Step 1 → INFINITE LOOP
Step 9: Browser gives up after ~20 redirects
         Shows "too many HTTP redirects" error page
```

### The Fix

**File**: `src/middleware.ts`

```ts
// BEFORE (broken):
if (currentTime >= exp) {
  return await handleLogout();  // calls backend, doesn't clear cookie
}

const handleLogout = async () => {
  try {
    await fetch(process.env.NEXT_PUBLIC_BACKEND + "/logout", { ... });
    // if backend is down, we get here:
  } catch (error) {
    // cookie never cleared!
    return NextResponse.redirect(new URL("/sign-in", req.url));
  }
};

// AFTER (fixed):
if (currentTime >= exp) {
  const response = NextResponse.redirect(new URL("/sign-in", req.url));
  response.cookies.set("jwt", "", { maxAge: 0, path: "/" });
  return response;
}
```

**What changed**:
- Removed `handleLogout()` function entirely — no more backend fetch in middleware
- When token is expired: immediately set cookie to `""` with `maxAge: 0` (deletes it), then redirect
- Same fix for the `catch` block (JWT decode errors)
- Cookie is always cleared before redirect → loop is impossible

Also cleaned up:
- Removed all commented-out code (old redirect URLs, commented validate-token calls)
- Removed redundant `console.error("JWT token is not available.")` for unauthenticated users
